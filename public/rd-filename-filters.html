Complete List of Real-Debrid's Filename Filters

I tested several thousand scene releases against Real-Debrid's torrent upload filter to map exactly which filename patterns trigger the infringing_file rejection (error_code 35). RD checks only the filename inside the .torrent metadata, it never downloads or inspects the actual file content.

TL;DR: Two Pattern Types, Every Current Release Format Affected

Type 1: 5 substring matches (case-insensitive, anywhere in filename):

web-dl   webrip   bdrip   hdrip   dvdrip

Type 2: Source.Codec dot-adjacency matches (case-insensitive):

BluRay.x264   HDTV.x264   HDTV.XviD   WEB.x264   WEB.h264   WEB.H264

Real-World Impact

From current PirateBay releases tested: 56 out of 165 filenames are blocked (~34%). The most impacted formats:

  • All [WEBRip] bracket releases, every single one blocked (Zootopia 2, Hoppers, Faces of Death, etc.)
  • All WEB-DL releases: AMZN/NF/DSNP/ATVP/HMAX/PMTP/iT/MA/TEN WEB-DLs all blocked
  • All .WEB.h264 scene releases: GRACE, ETHEL, EDITH groups all blocked
  • All .WEB.H264 releases: HiggsBoson, SYLiX groups blocked
  • All .HDTV.x264 releases: SYNCOPY group blocked

Safe formats: .WEB.h265, BluRay.x265, HEVC x265, TELESYNC, HDTS, DCP

Rule 1: Substring Matches

If the filename contains any of these strings (case-insensitive), the torrent is rejected immediately. Extension doesn't matter. Works in both dot-separated and space-separated names, and inside brackets.

Properties:

  • Extension-independent: .mp4, .mkv, .avi all checked
  • Works with spaces: 1080p WEBRip DDP5 1 x264 blocked
  • Works in brackets: [WEBRip], [BDRip] blocked

Rule 2: Source.Codec Dot-Adjacency Matches

When certain source tags appear immediately before certain codec tags with a dot separator, the torrent is blocked. Case-insensitive on both source and codec.

Key: WEB.H264 is blocked (capital H)

This was confirmed in run 4 with real releases: For.All.Mankind.S05E08.MULTI.1080p.WEB.H264-HiggsBoson and Chicago.Fire.S14E21.720p.WEB.H264-SYLiX both blocked. The match is case-insensitive on both source and codec.

Key: HDTV.h264 is NOT blocked

Despite HDTV.x264 and HDTV.XviD being blocked, HDTV.h264 and HDTV.H264 both pass. The HDTV rule only matches x264 and XviD, not h264.

Properties:

  • Only . (dot) triggers: BluRay-x264 ✅, WEB-h264 ✅, HDTV_x264 ✅
  • Must be adjacent: WEB.Line.H264 ✅ (Line breaks adjacency)
  • Order matters: x264.BluRay ✅
  • h265 / x265 / HEVC / AV1 never blocked in any source combo
